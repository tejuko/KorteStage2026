## Wat heb ik geleerd?
- **Double-serialization bug bij ACF**: bij `update_field()` via `wp eval` kan een waarde dubbel ge-serialized worden als je niet oplet. Eerst `get_field()` doen, kijken wat de werkelijke structuur is, dan pas updaten. Belangrijke valkuil voor batch-edits via WP-CLI.
- **Een nieuwe sectie bouwen op staging** is niet alleen markup + CSS: een gecombineerde animatie-choreografie (avatar wave + line sweep + roterende center-pill + 12 agent-pills) vraagt om timing-afstemming, niet alleen losse animaties.
- **Solid overlay + on-hover-naar-0** is een sterker UX-effect dan een gradient-overlay: gebruiker ziet meteen waar focus zit, en op hover komt de foto helemaal vrij.
- **Mobile is niet "desktop maar kleiner"**: voor de Senior Specialisten-sectie heb ik op mobile niet alles geschaald maar een eigen flow gebouwd (4 avatars i.p.v. 6, marquee i.p.v. statisch grid, kortere lijn, andere uitlijning).
- **Backup-pad doortrekken**: 5 verschillende backups van `style.css` op één dag = bewuste check-points tijdens een lange iteratie, niet alleen "voor het geval dat".
- **WordPress-hook-volgorde matters**: `wpautop` (10) → `do_shortcode` (11) → mijn preprocess (12). Eén priority-nummer fout en je shortcode komt vol met ongewenste `<br>` en `<p>`. Hier ga ik nooit meer toevallig overheen.
- **CSS scope-isolatie via prefix**: alle 2488 selectoren prefixen met `.rm-seo-calculator-wrap` was nodig om externe CSS (theme + andere plugins) niet te laten lekken in mijn calculator. Specificiteits-discipline = team-vriendelijke plugin.
- **CSS-lasagna is een real anti-pattern**: 15 lagen "even erbij prikken" werd onleesbaar. De "V16 clean reset" (alles eronder weghalen, één schoon blok eroverheen) is een nuttige techniek om vroeger uit te halen.
- **SCP silent fails** bestaan: een ssh-upload kan succesvol *lijken* maar inhoudelijk niets veranderd hebben. Altijd `grep` op de remote om te verifiëren dat de wijziging écht is geland.
- **Asset-permissies (chmod 600 vs 644)** kunnen de oorzaak zijn van "het laadt niet" — niet altijd JS of cache.
- Een **eigen plugin bouwen** in plaats van alles in het theme proppen is een fundamentele WordPress-architectuurkeuze. Plugin = herbruikbaar, los van theme; functionaliteit blijft staan bij theme-switch.
